## 1. Objectives

The goal of this project is to build and extend a simplified cache simulator so that you understand the most
important ideas behind cache hierarchy design:

```
how an address maps to a cache line, set, and tag
how block offset, set index, and tag are derived from cache geometry
how a cache access is classified as a hit or a miss
what should happen on a hit and on a miss
how a victim cache line is selected on replacement
how write-back and write-allocate affect behavior
how L1 and L2 caches form a hierarchy
how replacement policies and prefetchers affect hit rate and average memory access time
```
This project has **three tasks** :

1. Complete the core access logic for a single-level L1 cache.
2. Add an L2 cache below L1 and understand how a multi-level cache hierarchy works.
3. On a personalized trace distributed by the instructor, design suitable replacement policies and
    prefetchers to optimize hit rate and AMAT.

### What The Trace Analyzer Looks At

The script first parses every memory access in the trace and converts each address into a **block address** using
the cache block size.
It then computes several simple but useful statistics.

#### 1. Basic access mix

The analyzer prints:

```
total number of accesses
number and percentage of reads
number and percentage of writes
number of unique blocks touched
overall address range
```
This helps you answer questions such as:

```
Is the workload read-heavy or write-heavy?
Does it repeatedly touch a small footprint, or sweep across many distinct blocks?
```
#### 2. Block-stride distribution

The analyzer compares consecutive block addresses and reports the most common stride values.
For example:

```
stride 1 means the next access often goes to the next cache block
stride 2 means every other block is common
large positive strides may indicate regular jumps
negative strides may indicate backward scans or loop resets
```
This section is useful for deciding whether prefetching is likely to help.
If one small stride dominates, NextLine or Stride prefetching is usually worth testing.

#### 3. Hot blocks

The analyzer reports the most frequently accessed blocks.
If a small number of blocks appear very often, that usually indicates a **hot working set** or strong temporal
locality.

This can help explain why:

```
recency-based policies like LRU may work well
some data remains useful even when the trace contains other noisy accesses
```
#### 4. Hot L1 sets

The analyzer maps each block to an L1 set using the cache geometry and reports the most frequently used
sets.
This tells you whether accesses are spread out across the cache or concentrated into only a few sets.


If only a few sets dominate, that may indicate:

```
conflict pressure
set thrashing
a case where replacement policy matters more than raw cache size
```
#### 5. Per-window phase summary

The analyzer splits the trace into fixed-size windows and prints, for each window:

```
number of reads
number of writes
number of unique blocks
number of unique sets used in that window
```
This helps reveal whether the trace has multiple phases.
For example, one window might show strong reuse, while another looks like a streaming scan.
That kind of phase behavior is important when you explain why one policy helps in some regions but not
others.
